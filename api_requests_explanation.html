🔌 api_requests.py 文件详解

RAG 系统 LLM API 统一封装模块完整解析

🎯 文件的核心作用

api_requests.py 是 RAG 系统的LLM API 统一封装层，它提供了与多个大语言模型提供商（OpenAI、IBM、Google Gemini、阿里云 DashScope）交互的统一接口。

这个文件的主要职责包括：

  • 多提供商支持：统一封装 OpenAI、IBM、Gemini、DashScope 等多个 LLM API
  • 统一接口：提供一致的 API 调用接口，屏蔽不同提供商的差异
  • 结构化输出：支持 Pydantic Schema 的结构化输出
  • 错误处理：包含重试机制和 JSON 修复逻辑
  • RAG 集成：专门的方法用于 RAG 上下文问答
  • 批量处理：支持异步批量 API 请求

💡 为什么要写这个文件？

  1. 多提供商兼容性

    不同的 LLM API 提供商有不同的接口格式、参数命名和响应结构。这个文件统一封装，让上层代码无需关心底层实现细节。

  2. 代码复用和可维护性

    将 API 调用逻辑集中管理，避免在代码中多处重复实现，便于维护和更新。

  3. 统一错误处理

    集中处理 API 错误、JSON 解析错误、重试逻辑等，提高系统健壮性。

  4. 灵活切换提供商

    通过简单的参数切换即可使用不同的 LLM 提供商，便于测试和成本优化。

  5. 结构化输出支持

    统一处理结构化输出（Pydantic Schema），确保返回数据格式一致。

📚 主要类结构

1. APIProcessor（统一接口类）

作用：根据 provider 参数选择对应的基础处理器，提供统一的 API 接口

  • __init__(provider)：初始化，选择对应的基础处理器
  • send_message()：发送消息到 LLM（路由到对应的基础处理器）
  • get_answer_from_rag_context()：从 RAG 上下文获取答案
  • get_rephrased_questions()：问题重写（用于比较类问题）
  • _build_rag_context_prompts()：根据 schema 构建 RAG 提示词

2. 基础处理器类

BaseOpenaiProcessor

提供商：OpenAI

默认模型：gpt-4o-2024-08-06

特点：

  • 支持结构化输出（beta API）
  • 使用 OpenAI Python SDK
  • 自动统计 token 使用量

BaseIBMAPIProcessor

提供商：IBM Watson

默认模型：meta-llama/llama-3-3-70b-instruct

特点：

  • 支持余额查询
  • 支持嵌入生成
  • JSON 修复和重解析
  • 需要手动验证结构化输出

BaseGeminiProcessor

提供商：Google Gemini

默认模型：gemini-2.0-flash-001

特点：

  • 自动重试机制（最多3次）
  • JSON 修复和重解析
  • 支持模型列表查询

BaseDashscopeProcessor

提供商：阿里云 DashScope

默认模型：qwen-turbo-latest

特点：

  • 支持 Qwen 系列模型
  • 自动解析 JSON 响应
  • 兼容其他提供商的返回格式

🔧 核心方法详解

1. send_message() - 统一消息发送接口

def send_message( self, model=None, // 模型名称 temperature=0.5, // 温度参数 seed=None, // 随机种子（用于确定性输出） system_content="You are a helpful assistant.", // 系统提示词 human_content="Hello!", // 用户消息 is_structured=False, // 是否结构化输出 response_format=None, // Pydantic Schema（结构化输出时） **kwargs )

功能：统一的 API 调用接口，路由到对应的基础处理器

2. get_answer_from_rag_context() - RAG 上下文问答

def get_answer_from_rag_context( self, question, // 问题文本 rag_context, // RAG 检索到的上下文 schema, // 答案类型（name/number/boolean等） model // 使用的模型 )

功能：

  1. 根据 schema 选择对应的 Prompt 类
  2. 构建系统提示词和用户提示词
  3. 调用 LLM API 获取结构化答案
  4. 处理返回结果，确保格式一致

3. _build_rag_context_prompts() - 构建 RAG 提示词

def _build_rag_context_prompts(self, schema): // 根据 schema 参数选择对应的 Prompt 类 if schema == "name": system_prompt = prompts.AnswerWithRAGContextNamePrompt.system_prompt response_format = prompts.AnswerWithRAGContextNamePrompt.AnswerSchema elif schema == "number": // ... 其他类型 return system_prompt, response_format, user_prompt

4. get_rephrased_questions() - 问题重写

功能：将比较类问题拆解为针对每个公司的独立问题

// 示例：将 "哪家公司营收更高？" 拆解为两个独立问题 original_question = "2022年哪家公司营收更高，'苹果'还是'微软'？" companies = ["苹果", "微软"] // 返回： { "苹果": "苹果公司2022年营收是多少？", "微软": "微软公司2022年营收是多少？" }

🔄 调用流程

步骤 1: 初始化 APIProcessor

processor = APIProcessor(provider="dashscope") // 根据 provider 选择对应的基础处理器

步骤 2: 调用 RAG 问答方法

answer = processor.get_answer_from_rag_context( question="中芯国际的CEO是谁？", rag_context=context, // 检索到的上下文 schema="name", model="qwen-turbo-latest" )

步骤 3: 构建提示词

system_prompt, response_format, user_prompt = processor._build_rag_context_prompts("name") // 从 prompts.py 获取对应的 Prompt 类和 Schema

步骤 4: 调用基础处理器

answer_dict = processor.processor.send_message( system_content=system_prompt, human_content=user_prompt.format(context=rag_context, question=question), is_structured=True, response_format=response_format )

步骤 5: 返回结构化答案

{ "step_by_step_analysis": "...", "reasoning_summary": "...", "relevant_pages": [58], "final_answer": "张三" }

🛡️ 错误处理机制

1. JSON 修复（IBM 和 Gemini）

当 LLM 返回的 JSON 格式不正确时，使用 json_repair 库尝试修复：

try: repaired_json = repair_json(content) parsed_dict = json.loads(repaired_json) validated_data = response_format.model_validate(parsed_dict) return validated_data.model_dump() except Exception: // 如果修复失败，尝试重新解析 reparsed = self._reparse_response(content, system_content)

2. 重新解析（Reparsing）

如果 JSON 修复失败，使用 LLM 自身来重新格式化响应：

// 使用 AnswerSchemaFixPrompt 让 LLM 重新格式化 JSON reparsed_response = self.send_message( system_content=prompts.AnswerSchemaFixPrompt.system_prompt, human_content=user_prompt.format( system_prompt=system_content, response=response ), is_structured=False )

3. 自动重试（Gemini）

Gemini 处理器使用 tenacity 库实现自动重试：

@retry( wait=wait_fixed(20), // 等待 20 秒 stop=stop_after_attempt(3), // 最多重试 3 次 ) def _generate_with_retry(self, model, human_content, generation_config): return model.generate_content(human_content, generation_config)

📝 总结

api_requests.py 是 RAG 系统中的关键组件，它通过：

  • ✅ 统一封装多个 LLM API 提供商，提供一致的接口
  • ✅ 简化调用，上层代码无需关心底层实现细节
  • ✅ 错误处理，包含 JSON 修复、重试等机制，提高系统健壮性
  • ✅ RAG 集成，提供专门的方法用于 RAG 上下文问答
  • ✅ 灵活切换，通过简单的参数即可切换不同的 LLM 提供商
  • ✅ 结构化输出，统一处理 Pydantic Schema，确保数据格式一致

核心价值：将复杂的多提供商 API 调用逻辑封装成简单易用的接口，使得 RAG 系统能够灵活、可靠地与各种大语言模型交互，同时保持代码的清晰和可维护性。
